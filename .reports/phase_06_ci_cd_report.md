# Phase 6 CI/CD Engineering Report

## Objective

Design and document a complete CI/CD pipeline that automates linting, testing, security scanning, building, and deployment for wikisites, with a focus on reliability, security, and rapid iteration.

---

## Deliverables

### 1. Pipeline Configuration (`pipeline_config.toml`)

Defined a 5-stage sequential pipeline with clear dependency ordering:

| Stage    | Steps                                            | Dependencies   | Failure Behavior             |
| -------- | ------------------------------------------------ | -------------- | ---------------------------- |
| Lint     | ESLint, Prettier Check, TypeScript Check         | None           | Blocks all subsequent stages |
| Test     | Vitest Unit, Vitest Integration                  | Lint           | Blocks build and deploy      |
| Security | npm audit, Snyk Test, CSP Check                  | Lint           | Blocks build and deploy      |
| Build    | Astro Build (encp), Astro Build (wiki)           | Test, Security | Blocks deploy                |
| Deploy   | Cloudflare Pages (encp), Cloudflare Pages (wiki) | Build          | Triggers rollback on failure |

**Key Design Decisions**

- Lint runs first because it is fastest and catches obvious issues early.
- Test and Security run in parallel after Lint to minimize total pipeline time.
- Build depends on both Test and Security passing, ensuring only validated code is built.
- Deploy is gated on all previous stages and triggered only on `main` branch or pull requests.

### 2. Deployment Strategy (`deployment_strategy.md`)

Comprehensive deployment workflow covering:

**Automatic Deployment**

- Production deployment from `main` branch after all CI stages pass.
- Preview deployment for every pull request with URL posted as PR comment.
- Staging environment for pre-production validation.

**Environment Management**

- Production: `encp.wikisites.dev` and `wiki.wikisites.dev` on Cloudflare Pages.
- Staging: `staging-encp.wikisites.dev` and `staging-wiki.wikisites.dev`.
- Preview: Per-branch URLs generated by Cloudflare Pages.

**Manual Approval Gates**

- Hotfix merges to `main` require maintainer approval.
- Infrastructure changes require maintainer approval.
- Major dependency bumps require maintainer approval.
- Feature merges are auto-approved when CI passes.

**Blue-Green Deployment**

- Cloudflare Pages inherently supports blue-green via deployment versioning.
- New deployment uploaded as "green"; traffic switched automatically.
- Previous deployment retained as "blue" for instant rollback.
- Rollback time: < 2 minutes.

**Rollback Procedures**

- Automatic rollback on post-deployment Lighthouse critical failure.
- Automatic rollback on error rate spike (> 50% increase within 10 minutes).
- Manual rollback via `wrangler pages deployment rollback` CLI.
- Full rollback process (detection to verification) target: < 30 minutes.

## Integration with Phase 5.5 Regression Detection

The CI/CD pipeline directly integrates with the regression detection system:

| Pipeline Stage | Regression Detection Integration                                 |
| -------------- | ---------------------------------------------------------------- |
| Lint           | Prevents formatting and type regressions                         |
| Test           | Coverage reports feed into coverage regression detection         |
| Security       | Vulnerability scans feed into security alerting                  |
| Build          | Build time and bundle size metrics compared against baselines    |
| Deploy         | Post-deployment Lighthouse audit validates performance baselines |

**Baseline Enforcement**

- Build stage extracts bundle sizes and compares against `baseline_metrics.toml`.
- Build stage records build time and compares against baseline.
- Post-deployment Lighthouse audit compares all scores against baselines.
- Any threshold breach triggers alerting per Phase 5.5 rules.

## Security Architecture

### Secrets Management

- All secrets stored in CI platform secret store, never in repository.
- Secrets scoped to minimum required permissions.
- Cloudflare API tokens limited to Pages project scope.
- Snyk token for vulnerability scanning.
- SMTP credentials for email alerting.

### Supply Chain Security

- `npm ci` used in CI for reproducible builds (not `npm install`).
- Lockfile committed and validated.
- Dependencies pinned to specific versions.
- Automated dependency updates via Renovate with maintainer review.
- Continuous vulnerability monitoring via Snyk.

### Deployment Security

- No direct SSH access to infrastructure.
- All deployments logged with full audit trail.
- Cloudflare API tokens rotated quarterly.
- Deployment requires passing all security checks.

## Performance Considerations

### Pipeline Execution Time

| Stage     | Estimated Time | Optimization                                           |
| --------- | -------------- | ------------------------------------------------------ |
| Lint      | ~30 s          | ESLint caching, incremental TypeScript                 |
| Test      | ~60 s          | Vitest watch mode for development, parallel test files |
| Security  | ~45 s          | npm audit fast, Snyk caching                           |
| Build     | ~90 s          | Astro incremental builds, shared chunks                |
| Deploy    | ~15 s          | Cloudflare Pages fast upload                           |
| **Total** | **~240 s**     | Matches baseline build time                            |

### Caching Strategy

- Node.js dependencies cached between CI runs.
- ESLint cache enabled for incremental linting.
- Astro build cache for unchanged pages.
- Vitest module cache for faster test execution.

## Risk Assessment

| Risk                             | Likelihood | Impact | Mitigation                                                      |
| -------------------------------- | ---------- | ------ | --------------------------------------------------------------- |
| Cloudflare Pages outage          | Low        | High   | Blue-green with rollback; monitor Cloudflare status             |
| CI platform outage               | Low        | Medium | Manual deployment as fallback; document procedure               |
| Secret compromise                | Low        | High   | Rotation quarterly; minimum privilege; audit trail              |
| Dependency supply chain attack   | Medium     | High   | Lockfile validation; Snyk monitoring; pinned versions           |
| Rollback failure                 | Very Low   | High   | Test rollback procedure quarterly; maintain previous deployment |
| Pipeline performance degradation | Medium     | Low    | Monitor pipeline times; adjust caching; optimize stages         |
