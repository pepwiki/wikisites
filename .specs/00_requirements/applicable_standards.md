## 1. Executive Summary

This document maps every applicable standard to the two oligopeptide educational websites: **encyclopeptide.com** (formal encyclopedic reference) and **wikipept.com** (collaborative wiki-style educational platform). For each standard, we identify the standard ID, full name, domain, applicable clauses, how it applies to these sites, verification method, priority, and evidence location.

Standards are grouped into ten domains: software life cycle, design description, test documentation, web accessibility, security, privacy, content standards, performance, internationalization, and deployment. A conflict resolution section establishes the priority matrix when standards impose contradictory requirements.

## 12. Standard Conflict Resolution

### 12.1 Priority Matrix

When standards impose contradictory requirements, the following priority matrix determines which standard takes precedence:

| Conflict Category                            | Higher Priority                                    | Lower Priority                                      | Resolution Rationale                                                                                                                                                                                                                                                                  |
| -------------------------------------------- | -------------------------------------------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Accessibility vs. Visual Design**          | WCAG 2.1 AA                                        | Brand identity guidelines                           | Legal compliance (accessibility) supersedes aesthetic preference. Accessibility violations create legal liability and exclude users. Visual design can be adapted without losing brand identity.                                                                                      |
| **Security vs. Usability**                   | OWASP Top 10 / NIST SP 800-53                      | UX convenience                                      | Security vulnerabilities can compromise user data and site integrity. Usability can be enhanced without sacrificing security. Zero-tolerance for OWASP Top 10 findings.                                                                                                               |
| **Privacy vs. Analytics**                    | GDPR / CCPA                                        | Analytics granularity                               | Privacy regulations are legally mandated. Analytics can be implemented with privacy-first approaches (Cloudflare Web Analytics) that provide sufficient insights without violating privacy.                                                                                           |
| **Performance vs. Feature Richness**         | Core Web Vitals thresholds                         | Feature completeness                                | Performance directly affects user experience, SEO, and accessibility. Features that degrade performance below thresholds must be optimized or deferred. Performance is a feature.                                                                                                     |
| **Standards Compliance vs. Speed to Market** | Must-have standards (WCAG, OWASP, Core Web Vitals) | Should-have standards (IEEE 1016, ICU, Dublin Core) | Must-have standards are non-negotiable for launch. Should-have standards can be progressively adopted post-launch. Could-have standards are aspirational.                                                                                                                             |
| **ISO/IEC 12207 vs. Agile Practices**        | ISO/IEC 12207 (tailored)                           | Pure Agile methodology                              | ISO 12207 provides the governance framework; Agile provides the execution methodology. Tailoring the standard to fit Agile practices is the recommended approach. Document lightweight artifacts that satisfy standard requirements without creating excessive overhead.              |
| **NIST SP 800-53 vs. OWASP Top 10**          | OWASP Top 10 (application-level)                   | NIST SP 800-53 (organizational)                     | OWASP Top 10 provides specific, actionable web application security controls. NIST SP 800-53 provides the organizational framework. At the application level, OWASP findings take precedence for immediate remediation. NIST controls are addressed through organizational processes. |
| **Section 508 vs. EN 301 549**               | EN 301 549 (broader scope)                         | Section 508 (narrower scope)                        | EN 301 549 is a superset of Section 508 requirements. Compliance with EN 301 549 inherently satisfies Section 508. If only U.S. market is targeted, Section 508 suffices.                                                                                                             |
| **GDPR vs. CCPA**                            | GDPR (stricter requirements)                       | CCPA (less restrictive)                             | GDPR provides broader and stricter privacy protections. Compliance with GDPR generally satisfies CCPA requirements. If both apply, GDPR standards are used as the baseline.                                                                                                           |
| **Schema.org vs. Dublin Core**               | Schema.org (search engine priority)                | Dublin Core (academic indexing)                     | Schema.org is prioritized for search engine visibility and rich results. Dublin Core is layered on top for academic interoperability. No direct conflict; they are complementary.                                                                                                     |
| **Cloudflare SLA vs. Performance Budgets**   | Performance budgets (user experience)              | Cloudflare SLA (infrastructure)                     | Cloudflare SLA guarantees infrastructure availability. Performance budgets guarantee user experience. Both must be met, but if Cloudflare infrastructure is available yet performance budgets are exceeded, the issue is in application design, not infrastructure.                   |

### 12.2 Conflict Resolution Process

1. **Identification:** During design review or testing, identify any conflicting requirements between standards.
2. **Classification:** Determine the conflict category using the priority matrix above.
3. **Analysis:** Assess the impact of each possible resolution on compliance, user experience, and technical feasibility.
4. **Decision:** Apply the priority matrix to select the resolution approach. Document the decision in an Architecture Decision Record (ADR).
5. **Implementation:** Implement the resolution and update relevant specifications.
6. **Verification:** Verify that the resolution satisfies the higher-priority standard without unacceptable degradation of the lower-priority standard.
7. **Documentation:** Record the conflict, analysis, decision, and rationale in `.adrs/`.

### 12.3 Mandatory vs. Aspirational Compliance

| Tier                                        | Standards                                                                                       | Compliance Level                               | Enforcement                                               |
| ------------------------------------------- | ----------------------------------------------------------------------------------------------- | ---------------------------------------------- | --------------------------------------------------------- |
| **Tier 1: Legal/Regulatory (Must)**         | WCAG 2.1 AA, GDPR, CCPA, Section 508, EN 301 549, NIST SP 800-53                                | Full compliance required before launch         | CI/CD gates, pre-launch audit, ongoing monitoring         |
| **Tier 2: Industry Best Practice (Must)**   | OWASP Top 10, Core Web Vitals, Cloudflare SLA, ISO/IEC 12207                                    | Full compliance required before launch         | CI/CD gates, automated testing, regular review            |
| **Tier 3: Professional Standards (Should)** | IEEE 1016, IEEE 829, ISO/IEC 27001, ISO/IEC 15288, ICU MessageFormat, CLDR, W3C Web Performance | Compliance targeted within 6 months of launch  | Progressive adoption, audit schedule, improvement backlog |
| **Tier 4: Enhancement (Could)**             | Dublin Core, Open Graph (partial), Schema.org extensions                                        | Compliance targeted within 12 months of launch | Aspirational, adopted as capacity permits                 |
